# SP3 — Performance Shell: Design Spec

**Date:** 2026-06-28
**Status:** Draft
**Sub-project:** SP3 of 6 (see decomposition doc)
**Tracks:** B (Surface)
**Depends on:** SP2 complete (vibes, warm-zone, guided overlay)
**Unlocks:** SP4

---

## Goal

Make Froola presentable and visually alive. SP2 made it worth playing — SP3 makes it worth
sharing.

Three additions:
1. **Landing page** — a proper hero screen at `/` so sharing the root URL doesn't drop people into a raw camera prompt
2. **Audio-reactive particles** — canvas particle system that pulses with sound
3. **Share button** — copies `/play` to clipboard with a transient confirmation

---

## Decisions locked

- **Routing via `react-router-dom`.** Two routes: `/` (landing) and `/play` (instrument). The `/play` URL is the canonical share link — it skips landing and goes straight to the camera prompt.
- **Particles are visual-only.** No new audio logic. Particle color and spawn rate are driven by the existing `commandRef` (tension) and `analyserRef` (amplitude) already threaded through the renderer in SP2.
- **All SP3 work is Track B.** Track A has no new deliverables — SP2 audio engine is stable and unchanged.
- **Landing page is static.** No canvas animation, no video preview, no live demo mode. Fast load, minimal complexity.

## Share button (`src/components/ShareButton.tsx`)

Fixed position, top-right corner of the play shell. Copies `/play` URL to clipboard.

```typescript
type Props = Record<string, never>;

export default function ShareButton(_: Props) {
  const [copied, setCopied] = useState(false);

  async function handleShare() {
    await navigator.clipboard.writeText(window.location.origin + '/play');
    setCopied(true);
    setTimeout(() => setCopied(false), 1500);
  }

  return (
    <button className="share-btn" onClick={handleShare}>
      {copied ? 'Copied!' : 'Share'}
    </button>
  );
}
```

Styling: semi-transparent dark pill, monospace font, matches `VibePicker` aesthetic.
Positioned: `position: fixed; top: 1rem; right: 1rem`.

## Audio-reactive particle system (`src/engine/renderer/particles.ts`)

### Data model

```typescript
type Particle = {
  x: number;       // canvas px
  y: number;       // canvas px
  vx: number;      // px/frame
  vy: number;      // px/frame
  life: number;    // 1.0 → 0.0
  decay: number;   // subtracted per frame
  radius: number;  // 2–5 px
  color: string;   // rgba, tension-mapped
};
```

### ParticleSystem class

```typescript
export class ParticleSystem {
  private particles: Particle[] = [];
  private readonly MAX = 60;

  spawn(
    x: number,
    y: number,
    amplitude: number,   // 0–1 from analyser
    tension: number      // 0–1 from commandRef
  ): void

  tick(ctx: CanvasRenderingContext2D): void
  // Updates positions, decays life, removes dead particles, draws survivors
}
```

### Spawn logic

Spawn rate: `Math.floor(amplitude * 3)` particles per frame (0 at silence, up to 3 at peak).
Spawn position: detected hand positions on canvas (or canvas center if no hands present).
Velocity: random radial direction, magnitude `1 + amplitude * 3` px/frame.
Decay: `0.015 + Math.random() * 0.01` per frame (~60–100 frame lifetime at 60fps).
Radius: `2 + Math.random() * 3` px.

### Color mapping

Same tension palette as the warm-zone:

```
tension 0.0 → rgba(245, 158, 11, life)   // amber
tension 0.5 → rgba(251, 100,  0, life)   // orange
tension 1.0 → rgba(99,  102, 241, life)  // indigo
```

Lerp between these three stops. Alpha = remaining `life` value.

### Draw order

Particles are drawn **after** the warm-zone background gradient and **before** the dial labels. This keeps them beneath the interactive UI layer.

## Renderer update (`src/engine/renderer/index.ts`)

SP3 adds the `ParticleSystem` to the existing render loop.

```typescript
export function useRenderer(
  canvasRef: RefObject<HTMLCanvasElement>,
  signalsRef: RefObject<GestureSignal[]>,
  analyserRef: RefObject<AnalyserNode | null>,
  commandRef: RefObject<MusicalCommand | null>   // from SP2
): void {
  // existing warm-zone, dial rendering ...

  // SP3: particle tick
  const particlesRef = useRef(new ParticleSystem());

  // In rAF loop, after warm-zone, before dials:
  const amplitude = readAmplitude(analyserRef.current);  // existing helper
  const tension = commandRef.current?.tension ?? 0;
  const hands = signalsRef.current.filter(s => s.present);
  const spawnX = hands.length > 0
    ? hands.map(h => h.x * w).reduce((a, b) => a + b) / hands.length
    : w / 2;
  const spawnY = hands.length > 0
    ? hands.map(h => h.y * h).reduce((a, b) => a + b) / hands.length
    : h / 2;

  particlesRef.current.spawn(spawnX, spawnY, amplitude, tension);
  particlesRef.current.tick(ctx);
}
```

export default function PlayShell() {

      <GuidedOverlay />                                {/* SP2 */}

}

## Done criteria

1. Navigating to `/` shows the landing page — no camera prompt
2. Clicking "Play →" navigates to `/play` and shows the instrument
3. Navigating directly to `/play` skips the landing page
4. Particles appear on canvas when hands are detected and audio plays
5. Particle spawn rate visibly increases with louder playing
6. Particle color shifts from amber toward indigo as tension rises (dim chord = blue-purple)
7. Share button copies `/play` URL; "Copied!" confirmation shows for 1.5s
8. All SP2 features (VibePicker, GuidedOverlay, warm-zone) still work on `/play`
9. `npm run build` — zero TypeScript errors
10. `npm test` — all tests pass

## Explicitly out of scope for SP3

Auth, Supabase, pricing, preset saving, MIDI export, Jam Mode, mobile-specific layouts,
advanced waveforms (SP5), animated canvas demo on landing page, OG/social preview images.

---

## Performance constraints

- Particle cap of 60 prevents rAF loop degradation
- `ParticleSystem.tick()` draws with a single `ctx.save/restore` per frame, not per particle
- Landing page loads with zero camera or MediaPipe initialization
