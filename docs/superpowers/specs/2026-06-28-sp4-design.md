# SP4 — Session Recording & Replay: Design Spec

**Date:** 2026-06-28
**Status:** Draft
**Sub-project:** SP4 of 6 (see decomposition doc)
**Tracks:** B (Surface)
**Depends on:** SP3 complete (routing, PlayShell, ShareButton)
**Unlocks:** SP5

---

## Goal

Let players record what they play, get a shareable link, and let anyone replay it — no backend, no auth, no storage costs.

---

## Decisions locked

- **URL-encoded storage only.** Recording is packed into base64url and embedded in the `/replay?d=…` URL. No Supabase, no server.
- **30-second cap.** Hard limit. Auto-stops at 30s.
- **Manual record.** User taps Record to start, Stop to end. No auto-capture rolling buffer.
- **10 samples/sec.** Fixed polling rate during recording.
- **All SP4 is Track B.** Replay calls the existing `engine.play()` — no new audio engine work.

---

## Data format

Each sample is **5 bytes**:

| Byte | Field | Range | Notes |
|------|-------|-------|-------|
| 0–1 | `dt` | uint16, ms | Time since previous sample (first sample: ms since record start) |
| 2 | `noteIdx` | uint8, 0–6 | Index into `NOTES = ['A','B','C','D','E','F','G']` |
| 3 | `qualityIdx` | uint8, 0–6 | Index into `QUALITIES` array |
| 4 | `vibe` | uint8, 0–3 | Current vibe index at time of sample |

`register` is omitted to save space — replay uses 0.5 default. Negligible musical loss.

**At 10 Hz for 30s:** 300 samples × 5 bytes = 1,500 bytes → ~2,000 chars base64url. Comfortable URL length.

**Encoding:** `Uint8Array` → base64url (no padding, `+→-`, `/→_`). Decoded on the replay page before playback.

## Recorder hook (`src/engine/recording/useRecorder.ts`)

```typescript
export type RecorderState = 'idle' | 'recording' | 'done';

export function useRecorder(
  signalsRef: RefObject<GestureSignal[]>,
  vibe: Vibe
): {
  state: RecorderState;
  elapsed: number;          // seconds, 0–30
  shareUrl: string | null;  // set when state === 'done'
  start: () => void;
  stop: () => void;
}
```

- Samples at 10Hz via `setInterval(100)` while recording
- Reads `signalsRef.current` each tick: derives `noteIdx` and `qualityIdx` from left/right hand positions using the same `pickIndex` logic as the coordinator
- Auto-stops at 30s
- On stop: encodes to base64url, sets `shareUrl = window.location.origin + '/replay?d=' + encoded`

```
● Rec              ← idle (dim red pill)

```

The component owns its own `useRecorder` call internally. Progress bar is a `position: fixed` `<div>` along the bottom edge, `width` driven by `elapsed / 30 * 100%`, red, 3px tall, z-index above canvas.

## Replay page (`/replay`)

**Route:** `/replay` — reads `?d=` query param on mount.

**Playback engine:** On mount, decode the recording. Walk samples using `setTimeout` chains (not `setInterval`) to fire each sample at the correct `dt` offset. For each sample:
1. Build a `MusicalCommand` from `noteIdx + qualityIdx` (same intervals table as SP2)
2. Call `engine.play(cmd)`
3. Update a `replayCommandRef` (drives warm-zone + particles via renderer)
4. Update a `replaySignalsRef` with synthetic left/right hand signals at the dial positions (drives dial highlight animation)

**Canvas:** Uses the existing `useRenderer` — same warm-zone, particles, dials. Orbs are suppressed (no real hand positions, just dial animation).

**UI overlays:**

```
┌──────────────────────────────────────────┐
│ ▶ playing…           [top-left badge]    │
│                                          │
│           [canvas / dials animate]       │
│                                          │
│  [on end]: ↗ Share this  |  Play →      │
└──────────────────────────────────────────┘
```

- `▶ playing…` badge fades out when playback ends
- End state: two buttons — "↗ Share this" (copies same replay URL) + "Play yourself →" (navigates to `/play`)
- If `?d=` is missing or malformed: show an error state ("This replay link is invalid.")

**Audio engine:** `ReplayShell` creates its own `AudioEngine` instance on mount (same pattern as `useCoordinator` in `PlayShell`). It calls `engine.resume()` on first user interaction (browser autoplay policy), then fires samples via `engine.play(cmd)`.

**Component:** `src/components/ReplayShell.tsx`

## Done criteria

1. Record button appears in play shell (bottom-left)
2. Tapping Record starts recording; elapsed counter and red progress bar appear
3. Tapping Stop (or 30s elapsing) stops recording; button becomes "↗ Share"
4. Tapping Share copies a `/replay?d=…` URL to clipboard ("Copied!" flash)
5. Opening that URL plays back the session: dials animate, warm-zone reacts, particles fire, audio plays
6. Playback ends naturally; "Share this" and "Play yourself →" appear
7. Malformed `?d=` shows error state
8. `npm run build` — zero TypeScript errors
9. `npm test` — all tests pass

## Explicitly out of scope for SP4

Auth, Supabase, replay history list, waveform visualisation of recording, editing/trimming, looping playback, MIDI export, Jam Mode.

---

## Performance constraints

- `encode` / `decode` must complete in < 5ms for 300 samples (no heavy crypto)
- `setTimeout` chain for playback — do not use `setInterval` (avoids drift on tab-switch)
- Replay canvas uses same rAF loop as play mode — no second loop
